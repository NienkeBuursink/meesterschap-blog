Kilian Valkhof

04-02-26

Html en css zijn declaratief, dus makkelijker te gebruiken dan js dat imperatief is. Html en css kunnen bijna net zoveel als js.
Het web blijft altijd werken. Je kan alleen nieuwe dingen toevoegen en leren. Wat je leert is niet de eindbestemming. Er blijven altijd ontwikkelingen. Dus check of het inmiddels beter kan.
Appearance: none zorgt voor lege div zonder styling van browser.
Input:focus-visible {} voor toegankelijkheid. Dit wordt alleen getoond als gebruiker met toetsenbord werkt.
Jquery gebruiken lijkt kort. Maar op de achtergrond wordt et nog veel gerund in jquery zelf. Dit kan makkelijker in css met scroll-behavior: smooth;. Moet met @media ( prefers-reduced-motion: no-preference )!! Zonder animatie is voor iedereen oké. Dus begin met reduced motion en doe dan een mediaquery voor no preference. Je bouwt een toegankelijke website en enhanced dit voor mensen zonder preference. Stukje mindset.
Voor een fixed header -> scroll-margin. Normale margin alleen als je scrolled
:target pseudo class voor animatie nadat je scriptie naar een target. URL met id van de target in html.
Live attribute (open in details) je kan het toevoegen. Maar daarna is het van de browser. Alleen de initial state. Daarna browser verantwoordelijk.
::marker pseudo element veranderd het pijltje van een summary tag. [open] summary::marker voor het pijltje naar beneden. Werkt ook voor list elements. summary -> maak duidelijk dat het een summary is door cursor of background bij de hover.
Dialogs voor popups ipv alert. Animatie kon eerst niet omdat er geen beginpunt was. In css kan je beginpunt definiëren met @starting-style {}
Baseline om te kijken welke features nieuw zijn.

Peter Paul Koch

19-02-26

Browsers hevveb hun eigen javascript engine. Deze behandelt de javascript code op verschillende manieren. Javascript beïnvloed de DOM. Die kan je in de js pluggen. AOM is bijna hetzelfde als DOM. Maar aom is specifieker voor assisted tech.
Kijk uit met verandering van layout. Dit is zwaar voor de browser. Bijv als je een extra kolom gebruikt. Rendering engine is voor verschillende browsets. Chromium/blink is de meest bekende.
Js met een defer, daarmee wordt eerst alle HTML ingeladen. Maar CSS heeft ook een parser die er lang over doet. Dan kom je HTML tegen zonder styling.
Defer vs async: zonder -> script meteen uitvoeren. Defer-> do it later. Async -> i dont care when, just not now.
Backward compatibility: Als iets ooit in een browser heeft gewerkt, blijft het altijd in de browser werken. Internet explorer 6 modus: haal de doctype weg
Microsoft met internet Explorer was de eerste browser die gratis was. Dat was revolutionair.
Konqueror belangrijke browser (2000) voor linux. Mac heeft deze gekozen als rendering engine want ze wilden anders zijn, maar wel makkelijk. Dit wordt webkit van safari
Flow(2020) ondersteunt weinig css. Goed om mee te testen.

Nils Binder

04-03-26

Works at 9elements -> communication design, product development, web development. Sectors work together closely.
CSS day is reaaally cool and i want to go! Talk about the 'boring' stuff (components and aligning). But it might not be that boring after all. First designers used photoshop. This was complicated to translate to code. Firma bridges the gap between design and development. Back when designers used photoshop, things werent always possible on the web. Now figma limits us, because much more is possible on the web.
Use fr in grids to distribute whitespace.
Animation: stop-motion steps on a background image, instead of videos. Subgrid reuses rows from the parent element, so things stay aligned when only one of the children changes size. You can use view transitions to change language.
All in all i think Nils is a great developer/designer. The websites he showed were amazing and showed some good technical teatures. They do, however, not seem that accessible to me. Especially with all the moving animations, the contrast between colors and fonts only using a thin outline. Ill have to keep in mind that no matter how cool a website can be, accessibility is the most important. The websites he showed were really inspiring though and i look forward to try some new creative stuff!

Robbert Broersma

12-03-26

NL design system. Formulieren zijn mega moeilijk. Maar tips om te helpen ->
Introtekst. Uitleg van het formulier, dus duratie, welke verplicht zijn, of het op te slaan is. Met wie wordt deze info gedeelt? Fouten verbeteren na submit? En dan met alerts.
Opslaan en later verder is super fijn. 'Helper' teksten zijn vaak te licht. Check contrast.
44x44 px is gebruiksvriendelijk (voor bijv. radiobuttons)
Fout ingevuld, niet alleen met kleur maar ook met een streep, of kruis.
Align gewoon alles links. Mensen met tunnelvisie hebben daar baat bij. Ookal zijn buttons rechts cooler.
Button met 'volgende' in een lijstje zetten als er nog iets onder zit. Dat is goed voor de screenreader.
Asterisk is niet altijd ideaal. Gebruik gw 'verplicht' Als daar ruimte voor is Als de stappen niet variabel zijn, zet er dan bij hoe ver de gebruiker al ism dus stap 1/4.
Input element linken aan tekst, bijv help text of error messages. Dat kan met aria-describedby. Kan ook op de hele fieldset gezet worden.
Knippen en plakken moet overal worden toegestaan. Dit is voor als typen lastig is. Of kortetermijn geheugen.
Aan het einde controleren of alle gegevens kloppen. Check bijv of google translate aan staat. bdi translate="no" (voorkomt vertalingen van inputs)
Niet meer width en height. Maar inline-size en block-size. Zo werken ze ook voor andere talen(die van rechts naar links schrijven of boven naar beneden)
Bevestigen dat iets is gelukt, of verwijzen naar vervolgstappen. Gelukt banner kan met een alert. Zo hoort de gebruiker meteen dat het goed is. Gebruik een duidelijke focus:visible
